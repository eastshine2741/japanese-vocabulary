# Lyrics Licensing — 분석과 의사결정

> 작성일: 2026-05-02
> 컨텍스트: 플레이스토어 출시 + 인앱 광고 모델 검토

## 1. 배경

이 앱은 J-POP 가사를 활용한 일본어 학습을 핵심 가치로 한다. 음원 재생은 YouTube MV 임베드로 처리되어 음원·영상 권리 측면은 YouTube의 KOMCA/JASRAC 포괄계약이 흡수한다. 그러나 **앱이 직접 가사 텍스트를 추출·표시·번역하는 행위**는 별도의 어문저작물 라이선스 영역이며, 이 문서는 그 처리 방식에 대한 분석과 의사결정을 정리한다.

목표:
- 플레이스토어 출시
- 인앱 광고 부착
- 향후 인앱 결제·유료화는 별도 검토

## 3. 가사 저작권 진단 — 이중 레이어

### (a) 원문 가사 — 어문저작물 복제·전송권
- LRCLIB/VocaDB 모두 가사 자체의 저작권 클리어런스를 보장하지 않는다. 이 앱이 가사 텍스트를 자체 서버에 저장·재배포하면 **앱 운영자가 침해 주체**로 식별된다.
- 한국 저작권법 §28(공표저작물 인용)은 "정당한 범위 + 원저작물 수요 대체 X"가 핵심 요건. 풀곡 가사를 한 화면에 펼치는 UI는 **"가사집"의 통상적 이용을 대체**하므로 인용 항변에 불리.

### (b) 한국어 번역 — 2차적저작물작성권
- 저작권법 제5조 ①항: "원저작물을 번역·편곡·변형·각색·영상제작 그 밖의 방법으로 작성한 창작물(이하 "2차적저작물"이라 한다)은 독자적인 저작물로서 보호된다."
- 저작권법 제22조: "저작자는 그의 저작물을 원저작물로 하는 2차적저작물을 작성하여 이용할 권리를 가진다."
- 영리 목적 번역은 §30(사적이용을 위한 복제) 예외에서 벗어남 → 원저작권자의 별도 허락 필요.

### (c) KOMCA 신탁 범위와 번역권
- **번역권은 KOMCA 신탁 범위에서 제외된다** (2차적저작물작성권 미포함).
- 즉 KOMCA 어문저작물 사용료를 내도 **번역본 제공 권리는 별도로 원저작권자/음악 출판사에게 받아야** 한다.
- 인디 규모에서 곡별 원저작권자 협상은 사실상 비현실적 → 번역 풀텍스트는 라이선스 정공법으로 풀기 어려움.
- 출처: [나무위키 — 한국음악저작권협회 §7.2](https://namu.wiki/w/%ED%95%9C%EA%B5%AD%EC%9D%8C%EC%95%85%EC%A0%80%EC%9E%91%EA%B6%8C%ED%98%91%ED%9A%8C) — "저작인격권(동일성 유지권)과 2차적 저작물 작성권(리메이크, 편곡, 번안 허락권 등)은 포함되지 않는다." (1차 출처 PDF는 추후 KOMCA에 직접 확인 필요)

### (d) 한일 상호관리계약
- KOMCA-JASRAC 2007년 체결. 일본 곡의 어문저작물 라이선스도 KOMCA 한 곳에서 처리 가능.
- 단 (c)의 번역권 한계는 일본 곡에도 동일하게 적용.

## 4. 시장 관행 vs 법조문

법조문상으로는 풀가사·풀번역 노출은 침해이나, **실제 단속은 매우 선택적**이다.

KOMCA·JASRAC 단속 우선순위(추정):
1. 방송·OTT·대형 스트리밍
2. 노래방·공연장
3. 대형 가사 사이트
4. 대형 모바일 게임 (음악 사용)
5. ...
6. 인디 학습 앱

소규모 학습 앱의 일반적 패턴:
- 자동 모니터링에 잘 안 잡힘 (가사 텍스트가 토큰화되어 크롤러 매칭 회피)
- 신고 들어오면 cease & desist → 합의 → 사후 사용료 산정
- 소송·앱 폐쇄까지 가는 일은 드묾

비교 사례: 한본냥의 일본어 세카이, 한음 일본어 등 한국 J-POP 학습 앱들이 다수 운영 중이며 공개된 단속 사례 없음. 다만 이들은 **풀가사 UI를 의도적으로 제공하지 않는다** — 단어장, 리듬게임, 문장 만들기 등 학습 게임 UI 위주.

## 5. 핵심 프레임: "가사집 대체 효과"

저작권 단속의 실질적 기준은 법조문이 아니라 **"이 화면이 가사 서비스(Bugs가사·Genius·Musixmatch 등)의 대체재인가"**이다.

판별 테스트: **"이 화면을 스크린샷으로 모으면 가사집이 되는가?"**

| 화면 형태 | 가사집 대체 | 단속 위험 |
|---|---|---|
| 풀가사 스크롤 페이지 | ✅ | 🔴 |
| 한 줄씩 흐르는 싱크 가사 (현재 라인만) | ❌ | 🟢 |
| 라인 단위 학습 카드 | ❌ | 🟢 |
| 단어장 + 예문 발췌 | ❌ | 🟢 |

한본냥/한음이 풀가사 UI를 빼는 것은 이 "가사집 분류"를 회피하기 위한 구조적 선택일 가능성이 높다. 풀가사 페이지가 있는 순간:
- 사용자가 스크린샷 모으면 그 자체로 가사집
- 권리자가 "학습 도구가 아니라 무허가 가사 서비스"라고 주장하기 쉬움
- §28 인용 항변의 "원저작물 수요 대체" 요건에 정면으로 걸림

## 6. 의사결정 — A안 채택 (Karaoke 모드)

### 채택안
**한 화면에 노출되는 라인 수를 항상 좁게 제한(슬라이딩 윈도우)하고, 풀가사 스크롤 페이지는 만들지 않는다.**

UX 정의:
- 윈도우 크기: 한 화면에 노출되는 라인 ≤ 2~3 (재생/일시정지 무관 동일)
- 재생 중: 윈도우가 재생 위치를 따라감
- **스크롤 → debounce → seek**: 사용자가 윈도우를 스크롤하면 디바운스 후 포커스된 라인 시각으로 MV seek (재생/일시정지 무관)
- 윈도우 외부의 라인은 화면에 동시 노출되지 않음 (스크롤 중 일시 노출되더라도 정지 시 윈도우 크기로 수렴)
- "전체 가사 보기" 버튼·진입점 없음

### 핵심 가드는 "윈도우 크기"이지 "일시정지 시 동작 제한"이 아니다
초기 검토에서 "일시정지 시 스크롤 비활성"을 가드로 고려했으나 폐기. 이유:
- 재생 중에도 흐르는 라인을 사용자가 읽을 수 있어 "정적 읽기 차단" 효과가 실질적이지 않음
- 마음먹은 사용자는 재생을 누르고 스크롤하면 됨 — 의미 있는 마찰이 아님
- 가사집 분류 회피의 실질적 결정 요인은 **"한 화면에 동시 노출되는 라인 수"** 이고, 이는 재생/일시정지와 무관하게 윈도우 크기 자체에서 나옴
- 구현 단순성·UX 매끄러움도 이 결정의 부수 이득

따라서 **재생/일시정지 무관하게 스크롤 seek 동작**한다.

### 4가지 학습 목표 충족도
1. **문장 단위 학습 + 품사 색 토큰** — ✅ 유지 (현재 라인 카드 안에서 그대로 작동)
2. **플래시카드 → 예문 → Player 점프** — ✅ 유지 (해당 라인이 현재 라인으로 표시되도록 seek)
3. **곡의 번역·언어유희 학습** — △ 부분적 (사용자가 노래에 맞춰 흐르는 번역을 따라 읽음. 전곡을 한눈에 음미하긴 어려우나 핵심 학습 가치는 유지)
4. **싱크 가사 따라부르기** — ✅ 유지

### 리스크 포지션
- 한본냥/한음과 동급 또는 그 이하의 단속 위험
- 백엔드 가사 캐시는 분석 데이터로 유지하되, 클라이언트 전송 시 윈도우로 잘라 보냄 → 풀가사 직배포 형태 회피
- 인앱 광고 부착 가능

### 보조 안전장치
- **약관/하단 고지**: "본 앱의 가사·번역은 학습 목적이며 저작권은 원권리자에게 있습니다. 권리자 신고 시 즉시 삭제합니다." + 신고 이메일
- **takedown 응답 프로세스**: 곡 단위로 빠르게 비공개 처리할 수 있는 백엔드 운영 도구
- **OSS 라이선스 화면**: Kuromoji(Apache-2.0), IPADic, UniDic(BSD-3-Clause), Gemini, YouTube 사용 고지
- **VocaDB 출처 표기** (백링크)

## 7. 구현 변경사항

### `app-rn/src/screens/PlayerScreen.tsx`
- `FlatList` 데이터를 `studyUnits` 전체가 아닌 **포커스 라인 ±N의 슬라이딩 윈도우**로 제한 (N ≤ 2~3)
- `initialNumToRender={studyUnits.length}` 제거
- 정지 상태에서도 한 화면에 윈도우 외부 라인이 동시 노출되지 않도록 수직 가시 영역 자체를 윈도우 크기로 제한
- **스크롤 → debounce → seek**: 사용자가 윈도우 안에서 스크롤하면 디바운스(예: 200~300ms) 후 포커스된 라인의 `startTimeMs`로 `youtubeRef.current?.seekTo()` 호출 (재생/일시정지 무관)
- 정지/이탈 시: 윈도우 비움
- "전체 단어 담기" 버튼은 유지 가능 — 단어 리스트는 가사 자체가 아닌 분석 결과물
- 곡 진입 직후 placeholder: "재생을 시작하면 가사가 흐릅니다"

### `backend/.../LyricProcessingService` 등
- 곡 단위 가사 저장 자체는 유지 (분석/번역의 백엔드 캐시 용도)
- API 응답에서 풀가사 일괄 배포 형태를 제거하거나, 클라이언트가 이를 가사집 형태로 렌더하지 않도록 응답 구조 검토
- 영구 저장된 가사 데이터는 **권리자 신고 시 즉시 비공개 처리** 가능한 운영 토글 추가

### 신규 추가
- 약관 화면 / 하단 푸터 고지
- 권리자 신고 채널 (이메일 또는 폼)
- 곡 단위 비공개 처리용 어드민 엔드포인트

| 유료화·인앱 결제 도입 검토 | 라이선스 정리가 선행. 무라이선스 상태에서의 유료화는 손해 산정에 매우 불리 |

## 9. 결정에서 명시적으로 제외된 옵션

- **풀가사 페이지 그대로 출시** — 가사집 대체 효과로 단속 위험 가장 높음
- **사용자 BYO 라이브러리 단독 채택** — 법적으론 가장 깨끗하나 첫 학습 마찰이 커서 MVP 단계 채택 보류. 트래픽 성장 후 헤비 유저용 옵션으로 재검토
- **KOMCA 직접 계약 (지금)** — 인디 규모에서 운영 오버헤드·비용 부담. 트래픽 성장 후 검토
- **Musixmatch 상업 라이선스 (지금)** — 동일 사유로 보류
