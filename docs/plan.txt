- 즉 APPROVE하기위해 song 필요

- recommendation candidate에는 song_id 없다. APPROVE하면 API 또는 배치가 song 검색하고, song 검색안된게 있으면 fail

추천곡 상태머신
1. 일주일마다 도는 배치로 Apple RSS를 조회하여 song_recommendation_candidate를 PENDING 상태로 생성(100개)
2. 관리자가 어드민에서 직접 APPROVE 또는 REJECTED로 변경
3. candidate를 APPROVE로 전환 시, artist name과 title으로 exact match되는 songs row가 있어야함. 없다면 APPROVE로 전환 불가능
3-1. candidate의 artist name과 title으로 곡 분석(=songs, lyrics 생성 및 분석 파이프라인)을 트리거하는 기능을 함께 제공하여, songs가 검색되도록 한 후 APPROVE 재시도할 수 있게 한다.
4. APPROVE로 전환 시 검색된 song id가 연결된 recommendation을 PENDING 상태, order 0으로 생성
5. 관리자가 어드민에서 song, lyrics 검토 후 recommendation의 order 조정하여 PUBLISHED 상태로 변경
6. 유저앱에는 가장 최근 yearweek의 recommendation을 읽어서 (order ASC, created_at ASC)로 정렬하여 내려준다.

곡 분석 파이프라인 개편
1. /api/songs/analyze 또는 어드민에서의 곡 분석 요청 시 트리거된다.
2. 유저앱, 어드민에는 즉시 응답하고, analyzation_request 테이블에 row INSERT 또는 message queue에 메시지 발행한다. 유저앱은 응답 받은 후 로딩뷰를 띄운다.
3. 가사 조회 -> youtube MV 조회 -> lyric 분석 순서로 비동기 처리하며, youtube MV 조회 완료 시점에 song과 lyric을 생성하고 FCM으로 유저앱에 알려서 로딩뷰를 내리고 player screen을 띄운다.

어드민 작업

남은 결정사항
- 곡 분석 파이프라인을 배치 이용할것인지, message queue 이용할 것인지.
  - 이미 batch 사용중이므로, batch 유지하는 것이 변경이 적다는 장점
  - 그러나 30초마다 도는 batch는 리소스 낭비이며, scalable하지도 않으며, 곡 분석은 유저가 트리거하는 액션이므로 짧은 latency를 위해서라면 message queue가 적합
