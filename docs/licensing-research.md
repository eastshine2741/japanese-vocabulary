# 라이선스 / 저작권 리서치 요약

이 문서는 본 앱(노래 기반 일본어 학습)의 **플레이스토어 출시 + 인앱 광고** 목표를 전제로,
서드파티 의존성과 핵심 콘텐츠(가사)의 저작권 이슈를 정리한 것이다.

> 본 문서는 공개 자료 기반 일반 정보 정리이며, 법률 자문이 아니다. 출시 직전엔 저작권 변호사 검토를 받는 것을 권장.

---

## 1. 서드파티별 라이선스 정리

### 1.1 기술 스택 (문제 없음)

| 의존성 | 라이선스 | 상업 사용 | 의무 |
|---|---|---|---|
| Kuromoji | Apache 2.0 | OK | 라이선스 사본 + NOTICE 포함 |
| UniDic | GPL v2 / LGPL v2.1 / **BSD New** 트리플 | **BSD 선택** 시 OK | 저작권 고지 |
| IPADic | 자체 라이선스 | OK이지만 "illegal use 금지" 조항 존재 (Debian non-free) | 라이선스 전문 게재 |
| Gemini API | Google 약관 | **Paid tier** 필수 | EEA/UK/스위스 사용자 있으면 paid 필수 |

**조치:**
- UniDic은 BSD-3-Clause로 명시 사용
- 앱 내 OSS 라이선스 화면에 Kuromoji / IPADic / UniDic 라이선스 전문 게재
- Gemini API는 paid tier 사용

### 1.2 콘텐츠 소스 (회색지대)

| 의존성 | API 라이선스 | 콘텐츠 라이선스 | 위험도 |
|---|---|---|---|
| LRCLIB | MIT (오픈소스) | **가사 저작권 보장 없음** ("zero profit" 정신) | 매우 높음 |
| VocaDB | 무료 (백링크 + 캐싱 필수) | 곡별 상이 (CC가 많으나 곡별 확인 필요) | 중간 |
| iTunes Search API | 약관 존재 — promotional purpose만 | 메타데이터 표시 OK / 미리듣기·앨범아트 사용은 매우 제한 | 메타데이터 OK, promo content 위험 |

**iTunes Search API 핵심 제약:**
- 미리듣기·앨범아트는 "iTunes 콘텐츠 홍보 목적"에만 사용
- "Download on iTunes" 배지 인접 배치 + "provided courtesy of iTunes" 표기
- 미리듣기 스트리밍만 (다운로드/캐싱/sync 금지)
- **다른 상품·서비스 홍보 금지** ← 광고/유료화 시 충돌 가능
- Rate limit ~20 req/min

**조치:**
- iTunes Search API는 **메타데이터 검색 + Apple Music 링크 유도**로 한정 사용
- 앨범아트는 검색 결과 화면에만 / 학습 화면엔 미사용 권장

## 5. 비교 사례

### 5.1 한본냥의 일본어 세카이 / 한음 일본어
- 둘 다 풀가사 페이지보다는 **단어장 / 게임형 학습 UI** 중심
- "가사 뷰어"가 아닌 "학습 도구"로 포지셔닝
- 정확한 라이선스 처리 방식은 비공개 → §28 인용 항변 + 회색지대 운영 가능성 추정

### 5.2 Lirica (스페인어/영어/독일어)
- Sony 등 레이블과 **직접 라이선스 계약**
- 정공법

### 5.3 나무위키 — **본인 앱이 따라할 수 없는 모델**
- 운영 법인이 **파라과이 umanle S.R.L.** (2016년 이전, 한국 법 집행 회피 목적)
- 익명 사용자 업로드 UGC 호스팅 모델
- 한국에서 차단·고발 시도 계속되지만 집행 비효율로 운영 지속
- **본인 앱과의 차이:**
  - 본인은 한국 법인/개인 → 직접 노출
  - Play Store가 단일 takedown 채널 (KOMCA가 Google에 신고하면 즉시 내려갈 수 있음)
  - 정치적 인질 효과 0
  - **나무위키는 "합법"이 아니라 "집행 비효율 위치에 자리잡음"**

### 5.4 일반 인디 학습 앱 운영 패턴
대부분의 작은 한국 J-POP 학습 앱은 다음 단계로 굴러간다:

```
[클로즈드 베타]   [퍼블릭·무광고]   [출시·광고 ON]   [성장]
   거의 0           매우 낮음         낮음~중간        중간~높음
```

KOMCA가 작은 앱에 처음 접근할 때는 보통 **소송이 아닌 라이선스 협의 레터**.
그 시점에 정식화하면 폐쇄 없이 흡수 가능.

## 6. UX 대안 — "풀가사 페이지"를 대체할 패턴

핵심 원리: **페이지네이션이 아니라 "볼륨 + 맥락"이 위험을 결정한다.**
세션당 노출 가사 총량을 줄이거나, 가사가 "학습 도구의 부속물"로 보이게 하는 게 핵심.

### 실제로 위험을 줄이는 패턴

**(1) 단어 중심 뷰를 메인으로 (스니펫 동반)**
- 곡 진입 시 첫 화면 = "이 곡의 단어"
- 각 단어에 1~2줄 컨텍스트 스니펫만 노출
- 본인 앱의 `analyzed_content`(형태소 토큰 + 위치) 데이터로 자동 생성 가능
- 시각적으로 "단어장 페이지"
- 타임스탬프 없는 가사도 OK

**(2) 학습 세션 모드 (수동 진행, 풀가사 페이지 없음)**
- 진입점이 "이 곡 학습하기" (가사 보기 X)
- 절 단위로 1~2개씩 노출 → 인터랙션 → 다음 절
- 정적 풀가사 화면 부재
- "변형적 이용" 항변 강화
- 타임스탬프 무관

**(3) 모르는 단어 포함 절만 펼침**
- 학습한 단어로만 구성된 줄은 자동 접힘
- 사용자별 개인화된 차별 노출
- 변형적 이용 색채 강함

### 시그니처는 개선되지만 노출량은 비슷한 패턴

**(4) 노래방 모드 only (재생 중에만 가사 표시)**
- 정적 페이지 제거
- **타임스탬프 필수** → 본인 케이스에선 절반의 곡만 가능 → 메인 패턴 부적합

**(5) 절 단위 스와이프**
- 한 화면에 한 절. 좌우 스와이프
- 사용자가 끝까지 보면 풀가사 노출 동일 → 시그니처만 살짝 바뀜

### 추천 조합

```
메인 뷰:    [단어 중심 + 스니펫]   ← 곡 진입 시 첫 화면
        ↓
보조 뷰 1:  [학습 세션 모드]       ← "이 곡 학습 시작" 버튼
보조 뷰 2:  [노래방 모드]          ← 타임스탬프 있는 곡만, 재생 중만
```

**정적 풀가사 페이지를 앱 어디에도 두지 않음**이 핵심.

## 7. KOMCA 정식 계약 — 단계 분리

대부분 사람이 **KOMCA = 계약 = 큰 부담**으로 묶어 생각하지만 분리 가능하다.

| 행동 | 비용 | 시점 |
|---|---|---|
| 사용료 견적 문의 (전화/메일) | 무료, 30분 | 지금 가능 |
| 계약 체결 + 사용곡 매월 신고 | 협상비·운영부담 | 트래픽 확보 후 |

견적은 **약속 없이 숫자만** 받을 수 있다. 어문저작물 단독 요율은 음원 스트리밍보다 낮은 경향.

## 8. 추천 출시 플레이북

### Phase 1 — 베타 (0~3개월)
- 단어 중심 뷰 + 학습 세션 모드 구현 (풀가사 페이지 X)
- 클로즈드 베타 → 퍼블릭 무광고 출시
- 광고는 켜되 최소 빈도 (수익보다 노출 학습)
- 아래 "리스크 완화 7가지 장치" 적용

### Phase 2 — 트래픽 확보 (3~12개월)
- 사용자 1k~10k 구간
- 운영 데이터 누적 (실제 사용 곡, 빈도)
- KOMCA 견적 문의 (약속 없는 숫자만)

### Phase 3 — 정식화 (광고 ON 시점 또는 KOMCA 접촉 시)
- KOMCA 어문저작물 직접 계약 (한일 상호계약으로 일본 곡 포함)
- 또는 Musixmatch SDK 전환 (`LyricProvider` 인터페이스로 갈아끼움)
- 또는 하이브리드 (일본 곡: KOMCA / 국제 카탈로그: Musixmatch)

## 9. 리스크 완화 7가지 장치

회색지대 운영 시 분쟁을 흡수 가능하게 만드는 구조적 장치들. **합법으로 만들지는 못하지만 분쟁 시 결과가 달라진다.**

1. **`LyricProvider` 인터페이스 추출** — LRCLIB → Musixmatch / KOMCA 전환을 1 config 변경으로
2. **곡 단위 takedown 시스템** — 관리자 toggle + 권리자 신고 채널 이메일 (정보통신망법 §44-2 임시조치 의무 대응)
3. **출처 / 권리자 표기 항상 노출** — 가사 화면에 "가사 출처 / 작사자 / © 퍼블리셔 / YouTube 듣기 링크"
4. **"학습 도구" 포지셔닝** — 스토어 설명, 약관, UI 모두 일관되게 (가사 뷰어가 아님)
5. **가사 텍스트 영구 캐싱 최소화** — 가능하면 매 요청 fetch (분석 결과만 저장)
6. **가사 화면에 광고 미배치** — 광고는 홈/플래시카드 등 별도 화면에만
7. **출시 주체 = 1인 법인 또는 사업자등록** — 개인 자산 직접 노출 회피

## 10. 핵심 의사결정

### 선택지 A — 정공법 (지금 라이선스)
- KOMCA 또는 Musixmatch 견적 받고 계약
- 풀가사 UX 합법적으로 운영
- 인디 단계엔 비용 부담

### 선택지 B — 단계적 정식화
- 단어 중심 / 학습 세션 모드로 출시 (§28 인용 항변 가능)
- 트래픽 확보 후 라이선스 정식화 시 풀가사 기능 해금
- **현재 권장 경로**

### 선택지 C — 의식적 회색지대
- 풀가사 + 라이선스 없이 운영
- 위법임을 인지하고 7가지 완화 장치 적용
- KOMCA 접촉 시 즉시 정식화 준비
- 한본냥/한음 앱이 추정상 이 영역
- **이건 합법이 아니라 리스크 베팅**

### 선택지 D — 풀가사 포기 (영구)
- 단어 중심으로만 운영
- 가장 안전, UX 한계 명확

## 부록: 핵심 사실 정리

- **가사 = 어문저작물** — 음원·MV와 별개의 권리 (음원은 YouTube 임베드로 해결됨)
- **JASRAC ↔ KOMCA 상호관리계약 (2007)** — 일본 곡 가사도 KOMCA 한 곳에서 처리 가능
- **§28 공표저작물 인용** — "정당한 범위·공정한 관행·교육 목적·출처 명시" 충족 시 항변 가능, 풀가사 표시는 "정당한 범위" 인정 어려움
- **나무위키 모델은 따라할 수 없음** — 파라과이 법인 + UGC 호스팅 + 정치 인질 효과 조합 필요
- **실제 인디 앱 단속 우선순위는 낮음** — 작은 앱 → 묵인 → 규모 커지면 라이선스 협의 레터 → 정식화 협상 단계로 진행
- **광고 ON 시점이 위험 임계점** — KOMCA 모니터링 우선순위가 영리 명확화 시점에 올라감
